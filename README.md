# Fundamentals-of-Data-Engineering

# Оценка исходных систем данных

## Набор вопросов инженера данных


### 1. Характеристики источника данных
**Каковы основные характеристики источника данных? Это приложение? Рой IoT устройств?**

Определение природы источника данных критически важно, так как от этого зависят все дальнейшие архитектурные решения.

| Тип источника | Характеристика | Примеры |
|---------------|----------------|---------|
| **Веб/мобильное приложение** | Высокий RPS, события от пользователей, REST/GraphQL API | Онлайн-банк, маркетплейс |
| **IoT устройства** | Огромное количество источников, нестабильное соединение, данные с задержкой | Умный дом, промышленные датчики |
| **ERP/CRM система** | Медленные изменения, реляционные данные, сложная бизнес-логика | 1С, SAP, Salesforce |
| **Логи сервисов** | Полуструктурированные, огромные объемы | Nginx, микросервисы |
| **Файловые системы** | Периодическая загрузка, разнородные форматы | CSV, Parquet, JSON |


### 2. Хранение данных
**Как хранятся данные в исходной системе? Сохраняются ли данные в течение длительного времени или они носят временный характер и быстро удаляются?**

| Тип хранения | Характеристика | Риск для DE | Действие |
|--------------|----------------|-------------|----------|
| **Постоянное хранение** | Годы, десятилетия | Низкий | Можно перечитывать, строить историю |
| **Ограниченное TTL** | 30–90 дней | Средний | Настроить регулярные выгрузки |
| **Временное хранение** | Часы–дни, удаляется после обработки | Высокий | Обязательно настроить CDC |


### 3. Скорость генерации данных
**С какой скоростью генерируются данные? Сколько событий в секунду? Сколько гигабайт в час?**

| Нагрузка | Событий/сек | Объем/час | Инструменты |
|----------|-------------|-----------|-------------|
| **Низкая** | < 1 000 | < 1 ГБ | PostgreSQL, обычный ETL |
| **Средняя** | 1k–10k | 1–10 ГБ | Airflow, Batch |
| **Высокая** | 10k–100k | 10–100 ГБ | Kafka, Spark Streaming |
| **Экстремальная** | > 1 млн | > 100 ГБ | Flink, ClickHouse |


### 4. Согласованность данных
**Какого уровня согласованности могут ожидать инженеры данных от выходных данных?**

| Уровень согласованности | Описание | Где встречается |
|-------------------------|----------|-----------------|
| **Строгая (ACID)** | Транзакционная консистентность | PostgreSQL, банковские системы |
| **Согласованность в итоге** | Данные консистентны через некоторое время | Распределенные системы, NoSQL |
| **Низкая** | Возможны расхождения, данные требуют проверки | Логи, IoT, внешние API |


### 5. Качество данных
**Если вы выполняете проверку качества выходных данных, то как часто встречаются несоответствия?**

Типовые проблемы качества:

| Проблема | Частота | Действие |
|----------|---------|----------|
| NULL в обязательных полях | 1–5% | Заполнить значением по умолчанию |
| Дубликаты | 0.1–1% | Дедупликация на уровне ETL |
| Неверный формат | 0.5–2% | Парсинг и очистка |
| Выбросы | 1–3% | Отфильтровать или пометить |


### 6. Дубликаты данных
**Будут ли данные содержать дубликаты?**

| Уровень дедупликации | Решение | Пример |
|----------------------|---------|--------|
| **На источнике** | Запрос к бизнесу, исправление на уровне источника | Уникальные ключи в БД |
| **В сырых данных** | SELECT DISTINCT или оконные функции | Ежедневные инкременты |
| **На уровне ETL** | row_number() или дедупликация в Spark | Обработка дублей в пайплайне |
| **В витринах** | GROUP BY с агрегацией | Уникальные записи для отчетов |


### 7. Задержка данных
**Будут ли некоторые значения данных поступать с опозданием?**

| Сценарий | Причина задержки | Решение |
|----------|------------------|---------|
| **IoT-устройства** | Нестабильное соединение | Watermark, late-data handling |
| **Распределенные системы** | Сетевые задержки | Окна с допустимой задержкой |
| **Пакетная загрузка** | График выгрузки не совпадает | Обработка late-arriving данных |
| **Мобильные приложения** | Офлайн-режим, синхронизация | Буферизация на устройстве |


### 8. Схема данных
**Какова схема поглощаемых данных? Придется ли соединять несколько таблиц или систем?**

| Уровень | Описание | Сложность |
|---------|----------|-----------|
| **Одна таблица** | Один источник, плоская структура | Низкая |
| **Несколько таблиц в одной БД** | JOIN внутри одной системы | Средняя |
| **Несколько разных БД** | Разные системы, разные форматы | Высокая |
| **Разные типы систем** | SQL + NoSQL + API | Очень высокая |


### 9. Изменение схемы
**Если схема меняется (например, добавляется новый столбец), как это решается?**

| Практика | Описание |
|----------|----------|
| **Schema Registry** | Хранилище версий схем (Avro, Protobuf) |
| **Backward Compatibility** | Новые поля добавляем как NULLABLE |
| **Forward Compatibility** | Старые читатели могут читать новые данные |
| **Evolution** | Плавное изменение, уведомление всех заинтересованных |


### 10. Частота извлечения
**Как часто следует извлекать данные из исходной системы?**

| Режим | Частота | Использование |
|-------|---------|---------------|
| **Real-time** | < 1 секунда | Критические системы, фрод-мониторинг |
| **Micro-batch** | 1-5 минут | Онлайн-отчеты, дашборды |
| **Batch (частая)** | 15-60 минут | Операционные отчеты |
| **Batch (дневная)** | 1 раз в день | Бизнес-отчетность |
| **Batch (редкая)** | Раз в неделю/месяц | Архивные данные, аналитика |


### 11. Снимки vs CDC
**Данные предоставляются в виде периодических снимков или событий обновления от системы CDC?**

| Аспект | Снимки (Snapshot) | CDC (Change Data Capture) |
|--------|-------------------|---------------------------|
| **Что передается** | Вся таблица целиком | Только изменения |
| **Объем данных** | Большой (полная таблица) | Маленький (только дельта) |
| **Частота** | Обычно 1 раз в день | Непрерывно, < 1 сек |
| **Нагрузка** | Высокая | Низкая |
| **Историчность** | Только текущее состояние | Полная история изменений |


### 12. Поставщик данных
**Кто/что является поставщиком данных, передающим данные для последующего использования?**

| Тип поставщика | Описание | Контактное лицо |
|----------------|----------|-----------------|
| **Внутренняя команда** | Другая команда в компании | Team Lead, Tech Lead |
| **Внешний API** | Сервис-провайдер, партнер | Менеджер по интеграции |
| **Государственные органы** | Статистика, открытые данные | Менеджер по работе с госорганами |
| **Внутренняя система** | ERP, CRM, своя БД | DBA, Администратор системы |


### 13. Влияние на производительность
**Повлияет ли считывание из источника данных на его производительность?**

| Риск | Решение |
|------|---------|
| **Нагрузка на прод-БД** | Читать с реплики (read replica) |
| **Блокировки при чтении** | Использовать pg_dump или бэкапы |
| **Слишком частые запросы** | Настроить CDC, а не poll-запросы |
| **Большие SELECT-ы** | Постраничная выгрузка, инкрементальный подход |


### 14. Зависимости данных
**Есть ли у исходной системы зависимости от данных выше по потоку обработки?**

Вопросы для оценки зависимостей:

- Есть ли другие системы, от которых зависит этот источник?
- Какова надежность вышестоящих систем?
- Какие данные передаются между системами?
- Как часто меняются вышестоящие системы?


### 15. Проверка качества данных
**Осуществляется ли проверка качества данных на предмет несвоевременных или отсутствующих данных?**

Метрики качества:

| Метрика | Описание | Как проверять |
|---------|----------|---------------|
| **Задержка** | Время между событием и его появлением | max(ingestion_time - event_time) |
| **Полнота** | % записей, которые пришли | COUNT(*) / expected_count |
| **Своевременность** | Данные пришли в ожидаемое время | Сравнение с расписанием |
| **Объем** | Количество записей | Сравнение с историческими данными |


## Итоговый чек-лист

- [ ] **Характеристика:** Определен тип источника (приложение, IoT, БД, API, файлы)
- [ ] **Хранение:** Известен TTL данных (хранятся долго или удаляются)
- [ ] **Скорость:** Оценен RPS и объем данных (событий/сек, ГБ/час)
- [ ] **Согласованность:** Определен уровень согласованности (ACID / eventual)
- [ ] **Качество:** Оценен процент несоответствий (NULL, ошибки форматирования)
- [ ] **Дубликаты:** Определена частота дубликатов
- [ ] **Задержка:** Оценена возможная задержка данных
- [ ] **Схема:** Понятна структура данных и необходимость JOIN-ов
- [ ] **Изменения:** Есть процесс управления изменениями схемы
- [ ] **Частота:** Определен режим выгрузки (real-time / batch)
- [ ] **CDC:** Выбран подход (снимки или CDC)
- [ ] **Поставщик:** Известен владелец и контактное лицо
- [ ] **Производительность:** Чтение не влияет на прод
- [ ] **Зависимости:** Понятны вышестоящие системы
- [ ] **Мониторинг:** Настроена проверка качества данных
